HW 3 (CS 635)

MPI-based Programming and Communication Operations

Teams: Maximum 2.

Due Date: March 26, 2014 [4:29 PM EST]. There are 3 weeks to do this assignment and there are two parts.

Required Problems from Text: 4.6, 4.16 and 4.18 (30 points). Include answers within the submitted PDF that goes with Programming exercise.

Program : Game of Life Program (70 points)

Introduction:
The original Game of Life (GOL) was developed in 1970 by John Conway at the University of Cambridge. The game demonstrates that some simple local rules can lead to interesting large-scale behavior. The game is played on a periodic grid of cells (i.e. a toroidal mesh). Each cell can either be occupied by an organism or it can be unoccupied.

The game begins either with a random selection of occupied cells, or with a pattern that is read in from a file. From this initial generation, the next generation is calculated using the following birth, death, and survival rules:

  • If an occupied cell has 0 or 1 occupied neighbors, the organism dies due to loneliness.
  • If an occupied cell has 4, 5, 6, 7, or 8 occupied neighbors, the organism dies due to overcrowding.
  • If an occupied cell has 2 or 3 neighbors, the cell continues to be occupied, that is, the organism survives in the next generation.
  • If an unoccupied cell has exactly 3 occupied neighbors, then this cell will be occupied in the next generation, that is, an organism is born.

    • For more information: http://en.wikipedia.org/wiki/Conway%27s_Game_of_Life

    Project Description:

    The purpose of this programming assignment is to design, implement, and evaluate a MPI program that performs the two-dimensional Game of Life calculations.

    The parallel version where the calculations are divided among a set of processes may require you to examine ways in which the grid of cells could be decomposed into smaller parts. If necessary, you can review earlier class lectures where 1D,2D and 2D-cyclic decompositions were discussed.

    Input/Output
    Your program should take the command line arguments: the number of generations (G) to iterate and the size (N ) of the N × N Game of Life grid. The grid we will implement also wraps around like a torroidal mesh so the points do not fall off the two-dimensional grid.

    The output of your program should include the size of the Game of Life grid, the number of generations performed, the number of processors/blocks that you used, and a list of resulting live cell coordinates (x, y) at the end of the Game of Life simulation.

    Testing and Generation 1

    To start off your Game of Life simulation, you can employ a random number generator to create the initial life grid.

    To test your programs for correctness, you might also want to generate synthetic input that verifies some of the patterns that are common. See the linked Wiki article.

    Benchmarking
    Your task is to design and perform a series of experiments that measure the performance of your parallel implementations. You should determine the best combination of decomposition i.e., row, block, block-cyclic varying the game of life grid N from 4, 8, 16, 32, 64, 128, ..., (as large as you can go).

    The design of these experiments is up to you- it’s time to utilize all that you have learned this semester to find an optimal solution that performs the Game of Life simulation as quickly as possible on a parallel system.

    What you need to turn in

    1. The source files of your programs.
    2. A short writeup describing how you went about parallelizing the Game of Life program and how you designed your benchmarks. [1 Page]
    3. Timing results for your programs for the different benchmark configurations.
      You can compile these timing results into nice-looking charts. Include them in the above report.

    How to submit your programs/report

    1. Create a directory that has the following structure:
    2. gmu-user-name/qol/your-source-files-go-here
    3. gmu-user-name/report/the-pdf-of-your-report
    4. Make sure that each of your source file directories can be built by simply executing 'make'.
    5. Do not put any test files in the above directories.
    6. Create a tar file from the above directory structure and compress it. That is the file should be "gmu-user-name-.tar.gz"
    7. Upload that single file in the blackboard website by the due date.
    8. Feel free to copy the Makefile and modify it to include any extra files which you may have.
    9. Please submit reports in .pdf format, not .doc or .docx

    Evaluation Criterion

    1. [60%] Correctness of Parallel Code, Speedup Achieved,
    2. [40%] Benchmark Designed, Different Ideas Studied and Explanation of Results
